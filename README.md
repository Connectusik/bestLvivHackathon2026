# Warehouse Logistics

**Інтелектуальна платформа управління складською логістикою та ланцюгами постачання**

> Best Lviv Hackathon 2026 | Full-stack MVP | React + TypeScript

---

## Про проєкт

**Warehouse Logistics** — це комплексна система управління складською логістикою, побудована для реального бізнесу. Платформа об'єднує управління складами, маршрутизацію доставок, прогнозування попиту, розподіл запасів та трекінг замовлень в єдиному інтерфейсі.

Система розроблена з фокусом на **українську логістичну інфраструктуру** — 5 регіональних хабів (Київ, Львів, Дніпро, Одеса, Харків), реалістичні сценарії доставок (гуманітарна допомога, медичні поставки, промислове забезпечення) та повна українська локалізація інтерфейсу.

### Чому це круто

- **Не черговий TODO-лист** — це повноцінна логістична платформа з алгоритмами оптимізації маршрутів, прогнозуванням попиту та автоматичним розподілом запасів
- **Працює офлайн** — PWA з повною офлайн-підтримкою, черга дій синхронізується при відновленні з'єднання
- **Шифрування даних** — AES-GCM 256-bit для чутливої інформації (адреси, коментарі клієнтів)
- **Інтерактивні карти** — OpenStreetMap з OSRM маршрутизацією в реальному часі
- **Drag & Drop Kanban** — візуальне управління статусами замовлень
- **PDF-звіти** — генерація професійних звітів з підтримкою кирилиці
- **Dark Mode** — повна підтримка темної теми з автовизначенням системних налаштувань

## Ключові можливості

### Інтелектуальний розподіл запасів

Система аналізує 7-денну історію попиту, розраховує трендові фактори (прискорення/уповільнення) та пріоритетні множники для кожного складу. На основі цих даних алгоритм автоматично рекомендує оптимальний розподіл нових поставок між складами, враховуючи:

- Тижневий попит на кожен продукт по кожному складу
- Коефіцієнти тренду (зростаючий/стабільний/падаючий)
- Терміновість та пріоритет запитів
- Поточні залишки на складах
- Резервний рівномірний розподіл при відсутності історичних даних

### Прогнозування попиту

Модуль форкастингу використовує **експоненційне ковзне середнє (EMA)** для аналізу трендів з урахуванням:

- Сезонності по днях тижня
- 14-денної історичної вибірки
- Пріоритетного бустингу (термінові запити мають більшу вагу)
- Довірчих інтервалів (нижня/верхня межа)
- Класифікації тренду: зростаючий / стабільний / падаючий

### Оптимізація маршрутів

Алгоритм маршрутизації побудований на основі:

- **Nearest-neighbor heuristic** — початкове рішення
- **2-opt local search** — локальна оптимізація маршруту
- **Haversine formula** — точний розрахунок відстаней на сфері Землі
- Підтримка мульти-складських маршрутів
- Розрахунок кумулятивних відстаней та орієнтовного часу доставки

### Розширений план розподілу

Для великих замовлень система автоматично планує **split-доставки** з кількох складів:

- Зонування за відстанню: зелена (0–50 км), жовта (50–150 км), червона (>150 км)
- Скоринг кожного варіанту
- Аналіз економії коштів
- Інтеграція з доступністю товарів на складах

## Архітектура

### Управління станом

Система побудована на **Context API + React Hooks + localStorage**. Чотири незалежні контексти забезпечують чіткий розподіл відповідальності:

```
App
├── AuthContext        — автентифікація, ролі (admin / worker / client)
├── AppContext         — склади, запити, вантажівки, поставки, відвантаження + мутації
├── NotificationContext — toast-повідомлення та центр сповіщень
└── ThemeContext       — перемикання світлої/темної теми
```

### Персистентність

Всі дані зберігаються в `localStorage` через кастомний хук `useLocalStorage`. Дані переживають перезавантаження сторінки. Архітектура спроєктована так, що контекстні мутації можна безболісно замінити на `fetch()` виклики до реального бекенду — достатньо підмінити реалізацію в провайдерах.

### Code Splitting

Кожна сторінка завантажується лазі через `React.lazy()` з автоматичним retry-механізмом для застарілого PWA-кешу.

## Ролі та функціонал

### Admin — Центр управління логістикою

Адмін має доступ до повного набору інструментів для управління всією логістичною мережею:

#### Інтерактивна карта (`/admin/map`)
- OpenStreetMap з кольоровими маркерами: червоний (головний склад), синій (регіональні), оранжевий (запити), зелений (вибраний)
- OSRM-маршрутизація з відображенням відстані та часу
- Зони відстані з кольоровим кодуванням
- Кешування маршрутів для продуктивності
- Клік на маркер — детальна інформація

#### Управління запитами (`/admin/requests`)
- Таблиця з фільтрацією за пріоритетом (терміновий / критичний / підвищений / звичайний) та статусом
- Затвердження/відхилення запитів з призначенням складу
- Планування розподілу (одинична або split-доставка)
- Повний lifecycle: Очікує → Затверджено → В дорозі → Доставлено
- Призначення вантажівок
- Бічна панель розширеного плану розподілу
- Експорт у PDF: звіт по запитах, звіт розподілу, маніфест доставки

#### Kanban-дошка (`/admin/kanban`)
- Drag & Drop між колонками статусів
- Картки з кольоровим кодуванням пріоритету
- Відображення: ID, продукт, кількість, адреса, склад, дата
- Оновлення статусу в реальному часі

#### Розподіл та попит (`/admin/distribution`)
- Тижневий аналіз попиту по продуктах з трендовими факторами
- Рекомендації щодо розподілу поставок
- Графіки: тренди попиту, покриття запасів, прогноз
- Плани ребалансування між складами
- Результати оптимізації маршрутів

#### Статистика та аналітика (`/admin/statistics`)
- KPI-картки: загальна кількість запитів, доставлено, в дорозі, вантажівки
- Кругова діаграма розподілу статусів
- Розподіл пріоритетів
- Рівні запасів по складах (стовпчикова діаграма)
- Аналіз попиту по продуктах
- Кумулятивний тренд: запити vs. доставлено з часом
- Розподіл поставок по складах

#### Управління вантажівками (`/admin/trucks`)
- Повний CRUD для автопарку
- Статуси: доступний / на маршруті / неактивний
- Інформація: водій, телефон, локація, вантажопідйомність, призначений склад
- Модальні вікна для створення/редагування

#### Управління поставками (`/admin/supplies`)
- Створення нових поставок з автоматичним розподілом по складах
- Алгоритм smart-розподілу на основі попиту
- Історія всіх поставок з деталізацією розподілу
- Миттєве оновлення залишків на складах

#### Склади (`/admin/warehouses`)
- 5 складів: 1 головний (Київ) + 4 регіональних
- Додавання нових складів з валідацією координат
- Візуалізація рівнів запасів
- Графіки по продуктах (стовпчикові / лінійні / area)
- Тижневий тренд-аналіз

### Worker — Операції на складі

Працівник складу бачить інформацію, релевантну його конкретному складу:

- **Огляд складу** — деталі, кількість персоналу, вантажівки
- **Управління запасами** — таблиці поставок та відвантажень, пошук наявності товарів
- **Пошук найближчого запасу** — який склад має потрібний товар
- **Автопарк** — доступні вантажівки та їх статуси
- **Активні запити** — поточні запити на доставку
- **Термінові алерти** — сповіщення про високопріоритетні запити
- **Офлайн-синхронізація** — кнопка ручної синхронізації + індикатор з'єднання

### Client — Замовлення доставок

Клієнт може створювати та відстежувати свої замовлення:

- **Створення запиту на доставку** — вибір продукту, кількість, адреса з геолокацією, пріоритет, коментарі
- **Шифрування даних** — адреса та коментарі шифруються AES-GCM перед збереженням
- **Наявність на складах** — інформація про найближчий склад з потрібним товаром
- **Історія запитів** — таблиця всіх замовлень з трекінгом статусу
- **Офлайн-черга** — запити зберігаються локально та автоматично відправляються при відновленні з'єднання

## Моделі даних

### Warehouse (Склад)
```typescript
{
  id, name, type: 'main' | 'regular',
  latitude, longitude, address, description,
  products: ProductStock[]  // інвентар
}
```

### DeliveryRequest (Запит на доставку)
```typescript
{
  id, productNumber, quantity,
  address, latitude, longitude,
  comment, priority: 'normal' | 'elevated' | 'critical' | 'urgent',
  status: 'pending' | 'approved' | 'in_transit' | 'delivered' | 'rejected',
  createdAt, approvedAt,
  assignedWarehouseId, assignedTruckId,
  assignedWarehouses[],    // для split-доставок
  encryptedPayload         // зашифровані чутливі дані
}
```

### Truck (Вантажівка)
```typescript
{
  id, driverName, phone, location,
  status: 'available' | 'on_route' | 'inactive',
  warehouseId, capacity     // кг
}
```

### Supply (Поставка)
```typescript
{
  id, productNumber, quantity, createdAt,
  distribution: SupplyDistributionEntry[]  // розбивка по складах
}
```

## Безпека

- **AES-GCM 256-bit шифрування** — чутливі дані клієнтів (адреси, коментарі) шифруються через Web Crypto API
- **SHA-256 хешування** — перевірка цілісності даних
- **Маскування даних** — чутлива інформація маскується при відображенні
- **Зберігання ключів** — криптографічні ключі генеруються та зберігаються безпечно

## Офлайн-підтримка (PWA)

Система працює як Progressive Web App:

- **Service Worker** кешує всі статичні ресурси, тайли карт та маршрути OSRM
- **Офлайн-черга** — всі дії користувача (створення запитів, оновлення статусів) зберігаються в localStorage
- **Автоматична синхронізація** — при відновленні з'єднання черга обробляється автоматично
- **Візуальний індикатор** — статус з'єднання відображається в інтерфейсі
- **Ручна синхронізація** — кнопка примусової синхронізації для працівників складу

## UI/UX

- **Responsive Design** — mobile-first підхід, коректне відображення на всіх пристроях
- **Dark Mode** — автовизначення системних налаштувань + ручне перемикання
- **Кольорове кодування пріоритетів** — червоний (терміновий), оранжевий (критичний), жовтий (підвищений), зелений (звичайний)
- **Toast-сповіщення** — автоматичне зникнення через 4 секунди, типізація (success/error/warning/info)
- **Центр сповіщень** — історія всіх нотифікацій
- **Empty States** — інформативні порожні стани з підказками
- **Error Boundary** — graceful обробка помилок рендерингу
- **Semantic HTML + ARIA** — доступність для скрінрідерів

## Що робить цей проєкт особливим

Це не просто фронтенд з красивими кнопками. Це **робоча логістична платформа** з реальними алгоритмами під капотом:

1. **Математика, а не магія** — прогнозування попиту через EMA, оптимізація маршрутів через 2-opt, розподіл запасів через аналіз трендів. Кожен алгоритм можна пояснити і він дає реальний результат.

2. **Production-ready архітектура** — Context API з чітким розподілом відповідальності, lazy loading, error boundaries, PWA. Замініть localStorage на API — і це готовий продукт.

3. **Безпека з коробки** — шифрування чутливих даних, маскування при відображенні, хешування для цілісності. Не "потім додамо", а зараз.

4. **Офлайн-first** — логістика не зупиняється коли зникає інтернет. Система працює, накопичує дії, синхронізує при з'єднанні.

5. **Реалістичні сценарії** — українські міста, реальні відстані, типові логістичні задачі. Не абстрактний "Item A → Location B", а медичні поставки в Харків з пріоритетом "терміново".

6. **Повний цикл** — від створення запиту клієнтом через затвердження адміном, призначення вантажівки, відстеження на карті до фінальної доставки. Kanban, статистика, PDF-звіти — все в одному місці.
